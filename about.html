Status Posting System

About

This Status Posting System for a social networking site allows users to post their status and save it to a database table for later search and retrieval.

What special features have you done, or attempted, in creating the site that we should know about?
  • I added the custom CSS framework Skeleton to create a responsive, user-friendly interface. Today, if a web application doesn't work for mobile, it "doesn't work".
  • Buttons have been strategically colored to assist in guiding the user-flow.
  • Tables were used when displaying search results to neatly align status messages.
  • I made the assumption that the email address on the homepage had to be clickable, hence a user can easily create a new email.
  • I made the assumption that the default sharing type is 'Only Me'.
Which parts did you have trouble with?
  • When implementing the search result table, I noticed that status text would overflow the cell and cause issues when viewing on smaller device screens.
  • While testing against my own hosted database, I was unable to validate if a table with the given name already exists. I then discovered my server hosted a MariaDB database (not MySQL) and hence had a different information_schema structure. Switching to XAMPP during development and testing resolved this issue.
  • I had trouble casting the date into the correct format from input string to PHP date(), to MySQL-accepted YYYY-mm-dd format. After much testing, I learned via the strtotime() function documentation that: "Dates in the m/d/y or d-m-y formats are disambiguated by looking at the separator between the various components: if the separator is a slash (/), then the American m/d/y is assumed; whereas if the separator is a dash (-) or a dot (.), then the European d-m-y format is assumed. If, however, the year is given in a two digit format and the separator is a dash (-), the date string is parsed as y-m-d." (source: https://www.php.net/manual/en/function.strtotime.php)
What would you like to do better next time?
  • There is room for improvement when it comes to the modularisation of similar page content, such as header and page title, which could be defined in their own functions, and dynamically inserted using require_once() rather than hard-coded in each file. This would allow for changes to be made in one location from where they are then propagated out.
  • I would like to experiment with environment variables in external files rather than hard-coded values in multiple files. Coming from a JavaScript world, the principles of a 12-Factor App should be consistently applied across projects.
  • I would like for the sections of this About page to collapse (by heading) into an accordion-like view which can be expanded and collapsed as desired for simpler navigation and mobile-friendliness.
  • Use Parameter Binding when creating the SQL queries.
What references/sources you have used to help you learn how to create your website?
  • I referred to this question on Stack Overflow to learn how we extract multiple checkboxes from the posted form using $_POST[].
  • To better understand the database connection component of this system, I frequently referred to the lecture slides.
  • Any PHP unknowns were resolved thanks to the PHP section of W3Scbools.
What you have learnt along the way?
  • A key learning for me was the MySQL table validation. Previously, I had not paid close attention to validating a table's existence but it is definitely something worth checking, going forward.
  • Despite not being new to the world of HTML, CSS, and PHP, I discovered inefficiencies in maintaining duplicate pieces of code - as outlined under: What would you like to do better next time? - which led me to ask questions around code modularity in PHP.
  • I learnt that MySQL and MariaDB configure their information_schema differently, causing compatibility issues when checking for a given table's existence.
  • I experimented with posting an array of permission checkboxes and creating another table in the database to match posts with permissions via a one-to-many relationship. However, I limited the scope of this project to a single (denormalised) table in the end.
